## الفصل الثالث: الثغرات الأمنية ومنهجيات الاكتشاف المتقدمة

### 3.3 تطوير وتحسين عمل أدوات الفحص (Tool Enhancement Logic)
لم يتم التعامل مع أدوات الأمن السيبراني كأدوات جاهزة (Off-the-shelf)، بل قمنا بإعادة هندسة منطق عملها عبر واجهات برمجة مخصصة (Custom API Wrappers) لإضافة ميزات استخباراتية:

#### 3.3.1 هندسة محرك SQLMap: تعزيز الاستدلال واستمرارية الجلسات
تم تطوير واجهة SQLMap لإضافة طبقات ذكاء غير متوفرة في الأداة الخام:
1.  **تعزيز الاستدلال (Heuristic Augmentation):** قمنا ببرمجة محرك يحلل "فوارق الاستجابة" (Response Diff) وشذوذ "طول المحتوى" (Content-Length Anomalies) لاكتشاف الثغرات حتى لو لم تظهر رسائل خطأ صريحة من قاعدة البيانات.
2.  **استمرارية الجلسات (Session Persistence):** النظام يقوم بإنشاء مجلدات فريدة لكل هدف (`~/.sqlmap-sessions`) مما يسمح للأداة باستعادة نتائج الزحف (Crawl) ونقاط الحقن المكتشفة سابقاً، مما يسرع الفحوصات المتكررة بنسبة 40%.
3.  **الفحص الموجه (Targeted Focusing):** في حال وجود معاملات (Parameters) في الرابط، يقوم النظام بتوجيه الأداة للفحص المركز فوراً وإلغاء الزحف التلقائي لتوفير الوقت.

#### 3.3.2 هندسة محرك FFUF: المعايرة متعددة الأبعاد لبيئات SPA
تعد تطبيقات الصفحة الواحدة (SPA) كابوساً لأدوات الفحص التقليدية، وقمنا بحل ذلك عبر:
1.  **المعايرة الاستباقية (Multi-dimensional Calibration):** قبل بدء الفحص، يقوم النظام بإرسال طلبات عشوائية لإنشاء "بصمة مرجعية" تعتمد على (Size, Words, Lines). هذا يسمح باكتشاف الملفات الحقيقية حتى لو كان الموقع يعيد HTTP 200 لجميع الروابط (Soft-404).
2.  **فلترة "عناقيد الضجيج" (Noise Clustering):** طورنا خوارزمية تكتشف ما إذا كانت مجموعة كبيرة من المسارات تعيد نفس البصمة (مثل صفحات WAF أو Cloudflare)، فتقوم بحظرها آلياً لمنع إغراق لوحة التحكم بنتائج وهمية.

#### 3.3.3 هندسة محرك Nikto: التوحيد القياسي وتقليل الضجيج
تعاني Nikto من كثرة النتائج المكررة وغير الأمنية، وحللنا ذلك برمجياً:
1.  **التوحيد القياسي (Canonical Normalization):** استخدام تعابير نمطية (Regex) لدمج النتائج التي تختلف فقط في أرقام الجلسات أو المسارات الفرعية. فبدلاً من ظهور 50 تنبيه لنفس الثغرة، تظهر نتيجة واحدة موحدة.
2.  **مرشح الأهمية (Severity Filter):** قمنا بتنزيل درجة خطورة التنبيهات الخاصة بترويسات الأمان المفقودة (مثل X-Frame-Options) إلى "INFO" لتركيز انتباه المحلل على الثغرات القابلة للاستغلال الفعلي.

### 3.4 منهجية "ذكاء المطابقة" ومعالجة المخرجات (Intelligence Pipeline)
تمر النتائج بمحرك "الذكاء الهجين" الذي يقوم بـ:
1.  **المطابقة مع NVD:** البحث عن CVE ID المقابل لكل خدمة مكتشفة مع جلب وصف الثغرة الرسمي.
2.  **المطابقة مع Exploit-DB:** ربط الثغرة بأكواد استغلال حقيقية وتصنيف حالتها (Actively Used, Available, Unproven).

## الفصل الرابع: التصميم التنفيذي، الابتكارات، والمتطلبات الهندسية

### 4.1 مميزات النظام الهندسي المتكامل والتميز التنافسي
يتفوق نظامنا على المنصات المشابهة في النقاط التالية:
1.  **الذكاء الهجين (Hybrid Intelligence):** دمج مخرجات الأدوات الحية مع قواعد بيانات استخباراتية محلية (NVD + Exploit-DB) مما يعطي "حكماً نهائياً" مدعوماً بمصادر عالمية وليس مجرد تخمين من الأداة.
2.  **إدارة دورة حياة الفحص (Lifecycle Management):** القدرة على إيقاف (Pause) واستئناف (Resume) الفحوصات في أي وقت بفضل معمارية FastAPI Asynchronous، وهي ميزة نادرة في الأدوات المؤتمتة.
3.  **نظام التنبيهات الحسي (Scoreboard):** لوحة تحكم تعرض عدادات لحظية تتغير مع تقدم الفحص باستخدام تقنية TanStack Query Polling، مما يعطي تجربة مستخدم تفاعلية (Real-time Feedback).
4.  **عزل البيانات المحكم (Self-Fortified Security):** استخدام RLS لضمان أن كل مستخدم يعيش في "جزيرة معزولة" برمجياً داخل قاعدة البيانات.

### 4.2 التحديات التقنية الكبرى والحلول المبتكرة (Real-world Hurdles)
*   **التحدي: البحث في ملايين السجلات (The Million-Record Lookup):**
    *   **الحل:** قمنا ببناء فهارس SQLite متقدمة لملفات NVD المحملة، مما قلل وقت البحث عن CVE من 120 ثانية إلى 8 مللي ثانية.
*   **التحدي: الفحوصات الطويلة وانقطاع الاتصال:**
    *   **الحل:** تنفيذ معمارية "المهام الخلفية" (Background Tasks) حيث يستمر الفحص على الخادم حتى لو أغلق المستخدم المتصفح، مع آلية Reconcile لتصحيح حالة المهام المتعثرة عند إعادة التشغيل.

### 4.3 مصفوفة المتطلبات الوظيفية وغير الوظيفية الشاملة

#### أولاً: المتطلبات الوظيفية (Functional Requirements)
| المتطلب | الوصف التفصيلي والدقيق |
| :--- | :--- |
| **إدارة الأهداف** | إضافة وتخزين أهداف الفحص (URLs/IPs) مع دعم تصنيفها برمجياً. |
| **الفحص المصادق** | حقن (Auth Cookies) في أدوات الفحص للوصول للمناطق المحمية (Authenticated Scanning). |
| **تكامل Exploit-DB** | الربط الآلي بين الثغرة المكتشفة وكود الاستغلال المتاح لها لتقييم واقعية الخطر. |
| **إدارة دورة الفحص** | التحكم الكامل (Start, Pause, Resume, Stop) في العمليات الجارية. |
| **لوحة المعلومات** | عرض رسوم بيانية (Donut Charts) توضح توزيع المخاطر وحالة الاستغلال. |
| **إدارة المستخدمين** | نظام دعوات آمن (User Invitations) مع لوحة تحكم للمسؤول لمراقبة النشاط. |

#### ثانياً: المتطلبات غير الوظيفية (Non-functional Requirements)
| المتطلب | المعيار الهندسي والحل |
| :--- | :--- |
| **عزل البيانات** | تطبيق Row Level Security (RLS) لعزل بيانات كل مستخدم على مستوى قاعدة البيانات. |
| **الأداء الحركي** | تحديث واجهة المستخدم كل 5 ثوانٍ (Polling) دون الحاجة لإعادة تحميل الصفحة. |
| **الاستجابة** | زمن استجابة الـ API لطلبات البحث والعرض لا يتجاوز 1.5 ثانية تحت الضغط. |
| **القابلية للتوسع** | تصميم Decoupled Architecture يسمح بإضافة خوادم فحص (Agents) جديدة بسهولة. |
| **الأمان الهيكلي** | حماية جميع مسارات النظام بـ JWT وتشفير كلمات المرور بـ Argon2 أو Bcrypt. |

### 4.5 ميزات محتويات التقارير الأمنية المولدة
يولد النظام تقارير PDF احترافية لكل عملية فحص، وتتميز بـ:
1.  **الملخص التنفيذي (Executive Summary):** رسم بياني لمستوى الخطر العام وتوصية سريعة للإدارة.
2.  **دمج الأدلة (Evidence Consolidation):** تجميع مخرجات الأداة الخام في كتل برمجية (Code Blocks) منسقة لسهولة المراجعة.
3.  **جدولة الإصلاح (Remediation Schedule):** اقتراح زمني للإصلاح (Critical: 7 days, High: 30 days).
4.  **روابط الاستغلال:** وضع روابط مباشرة لـ Exploit-DB داخل التقرير ليتمكن المختبر من تجربة الثغرة يدوياً.
